이건 지금 한국에서 매우 유망한 사업입니다.
AI 독거노인 돌봄 디바이스 전체 아키텍처

아래는 **AI 독거노인 돌봄 디바이스(비대면 케어 + 원격진료 연계)**를 “제품/서비스로 바로 설
계 가능한 수준”으로 정리한 전체 아키텍처입니다. (디바이스·클라우드·앱·병원·보안/운영까지)




1) 목표와 설계 원칙

   •    현장 사용성: 노인은 앱보다 음성/버튼/표정(LED·스크린) 중심

   •    안전성: 응급·낙상·무활동을 로컬에서도 1차 판단(클라우드 장애 대비)

   •    프라이버시: “항상 듣는” 장치이므로 최소 수집·로컬 처리 우선

   •    의료 연계: 원격진료는 진료행위 주체(의료기관), 장치는 측정·중계·알림 역할로 분리

   •    운영성: OTA, 원격진단, 장애복구, 콜센터/관제 연동




2) 전체 아키텍처 개요 (E2E)

시스템 구성 블록

   1.   Home Device Layer (엣지)

   2.   Connectivity Layer (네트워크/프로토콜)

   3.   Cloud AI & Data Layer

   4.   Care Service Layer (돌봄/관제/알림)

   5.   Telemedicine Integration Layer

   6.   Apps & Portals (보호자/요양/의료진)

   7.   Security & Compliance Layer

   8.   Ops/Monitoring Layer




3) Home Device Layer (엣지) 상세

3.1 디바이스 하드웨어(권장 2안)

A안: 저전력 음성 중심 (ESP32-S3 + 외부 오디오 DSP)

   •    장점: 저렴/저전력/대량배포 용이

   •    단점: 고품질 STT/비전은 서버 의존
B안: 온디바이스 AI 강화 (Raspberry Pi / Jetson Orin Nano 등)

   •   장점: 프라이버시·오프라인 대응↑(낙상·호흡·키워드 로컬)

   •   단점: 원가/전력/발열/운영 난이도↑

3.2 필수 센서/IO

   •   Mic Array(2~4ch) + Speaker

   •   물리 버튼(“도움 요청/응급”) + 상태 LED/소형 디스플레이

   •   IMU(낙상/충격/움직임)

   •   (옵션) mmWave 레이더(호흡/재실/낙상) ← 고령자 케어에 강력

   •   (옵션) 카메라: 원격진료/보호자 요청 시만 ON (프라이버시 모드 기본 OFF)

   •   BLE(혈압계/혈당계/SpO2/체온계 연동)

   •   LTE/5G(옵션: Wi-Fi 불안정 지역용 eSIM 모델)

3.3 로컬 소프트웨어 구성(Edge Runtime)

   •   Wake Word 엔진(로컬)

   •   STT 라우팅: 로컬 키워드/짧은 명령은 로컬 처리, 상담/긴 대화는 서버

   •   Safety Rule Engine (로컬 1차 판단)

          o   무활동 N시간, 낙상 후보 이벤트, “살려줘/숨이 차/가슴이 아파” 등 위험 키워드

   •   Device State Machine

          o   정상/주의/경고/응급/네트워크 장애 모드

   •   BLE Medical Device Gateway

          o   측정 데이터 수집→암호화→전송

   •   Offline Fallback

          o   네트워크 끊겨도 응급버튼→SMS/전화 게이트웨이(가능하면)

   •   Privacy Switch

          o   마이크 상시 대기(키워드만), “녹음 저장 금지” 기본 정책




4) Connectivity Layer (통신)

   •   MQTT: 디바이스 상태, 센서 이벤트, heartbeat (경량/실시간)
   •   HTTPS/REST: 설정/정책/펌웨어/리포트 업로드

   •   WebRTC: 원격진료 음성·영상(옵션)

   •   BLE: 혈압/혈당/SpO2 기기 연동

   •   메시지 큐: Cloud 내부 이벤트(알림·관제 파이프라인)




5) Cloud AI & Data Layer

5.1 AI 서비스

   •   ASR(STT): 음성→텍스트 (스트리밍 가능)

   •   NLU/LLM Orchestrator

           o   의도분류: (대화/약복용/증상/응급/일상/기기제어)

           o   정책기반 라우팅: 응급은 “설명”보다 “즉시 조치” 우선

   •   TTS: 텍스트→음성(친근한 페르소나)

   •   Conversation Memory (선택적)

           o   개인 정보는 최소화, 동의 기반

   •   Risk Classifier

           o   증상 문장 + 생체데이터 + 패턴 → 위험도(낮음/중간/높음/응급)

포인트: 의료 영역은 “진단”이 아니라 **“위험 신호 탐지/권고/연결”**로 설계해야 안전합니다.

5.2 데이터 저장소

   •   Time-series DB: 혈압/혈당/SpO2/활동량/레이다 호흡 등

   •   Event Store: 낙상 이벤트, 무활동 경보, 응급버튼

   •   User Profile DB: 복약 스케줄, 보호자 연락망, 병원/요양기관 정보

   •   Audit Log: 누가 언제 무엇을 열람/변경했는지 (필수)

   •   (선택) Vector DB: 개인 대화 기억(민감하므로 최소)




6) Care Service Layer (돌봄/관제/알림)

6.1 룰 + 워크플로우(핵심)

   •   알림 단계화
           1.   디바이스가 사용자에게 1차 확인(“괜찮으세요?”)

           2.   미응답 시 보호자/요양보호사 알림

           3.   위험도 높음/응급 키워드 시 관제센터/119 연계(정책)

   •    콜 트리(Call Tree)

           o    보호자 1 → 보호자 2 → 요양기관 → 관제센터 → 응급

6.2 운영자 콘솔(관제)

   •    사용자별 상태(정상/주의/경고/응급)

   •    최근 24시간 활동/측정/대화 요약(민감정보 최소)

   •    디바이스 온라인/배터리/통신품질

   •    알림 처리 이력(확인/조치/종결)




7) Telemedicine Integration Layer (원격진료 연계)

7.1 원격진료 플로우(표준)

   1.   사용자가 “의사 연결” 또는 위험도 높음으로 자동 제안

   2.   사전 문진(Pre-triage): 증상 체크리스트 + 측정데이터 묶음

   3.   병원 시스템(원격진료 플랫폼)으로 예약/접수

   4.   WebRTC로 연결 (또는 병원 앱 딥링크)

   5.   처방/지도/추적관리

7.2 인터페이스

   •    병원/원격진료 플랫폼이 제공하는 API(없으면 중계 서버)

   •    의료 데이터는 표준 포맷(가능하면 HL7 FHIR) 고려

   •    처방·진료기록은 병원 시스템에 귀속, 장치는 “요약/알림”만




8) Apps & Portals

8.1 보호자 앱

   •    오늘 상태 요약(활동/수면/측정/알림)

   •    긴급 알림 수신/통화/영상 연결
   •   복약 스케줄 설정(권한 분리)

   •   “프라이버시 모드/카메라 요청” 같은 민감 기능은 이중 확인

8.2 요양기관/지자체 포털(B2G/B2B)

   •   대상자 리스트, 위험도 순 정렬

   •   방문 일정/케어 기록

   •   리포트 자동 생성(월간/주간)

8.3 의료진 포털(선택)

   •   측정 트렌드, 최근 증상 문진 결과

   •   원격진료 세션 이력(병원 플랫폼과 연동)




9) Security & Compliance Layer (필수)

   •   디바이스 인증: 기기별 인증서(Provisioning)

   •   전송 암호화: TLS 기본

   •   저장 암호화: KMS 기반(서버), 디바이스 저장 최소화

   •   권한/역할 분리(RBAC): 보호자/요양/의료진/운영자

   •   감사로그(Audit): 조회/수정/알림 조치 전부 기록

   •   데이터 최소수집 + 보관기간 정책

   •   민감 기능 보호: 카메라/대화 기록 열람은 강한 인증(MFA)




10) Ops/Monitoring (실제 서비스 필수)

   •   OTA 업데이트(롤백 포함)

   •   기기 원격 진단(마이크/스피커 테스트, 네트워크 품질)

   •   장애 알림(SRE): heartbeat 끊김, 배터리 급감, 센서 오류

   •   A/B 정책 배포(룰 변경, 알림 임계치 조정)

   •   모델 버전 관리(ASR/TTS/분류기)




11) “핵심” 이벤트 흐름 3가지 (현장에서 가장 중요)
1) 무활동/낙상 의심

  •    Edge 감지 → 사용자 확인 → 미응답 → 보호자/관제 → 필요 시 119

2) 급성 증상(호흡곤란/흉통 등)

  •    음성에서 위험 키워드 + (가능시) SpO2↓ → 즉시 경고 모드

  •    보호자/관제 자동 알림 + 원격진료 연결 제안

3) 복약/만성관리

  •    스케줄 알림 → 복용 확인 → 누락 시 보호자 알림(옵션)




12) MVP(최소기능) 추천 스코프

처음부터 원격진료 풀셋은 무겁습니다. MVP는 다음 5개만 잡아도 “돌봄 디바이스”로 시장 테스
트가 됩니다.

  1.   음성 대화 + 안부 체크

  2.   응급 버튼 + 콜 트리

  3.   무활동 감지(레이더 또는 IMU/마이크 기반)

  4.   BLE 혈압/SpO2 연동

  5.   보호자 앱(상태/알림/통화)




원하면 다음 단계로 바로 이어서 작성할 수 있습니다.

  •    (A) 아키텍처를 Mermaid 다이어그램으로 시각화

  •    (B) 디바이스 하드웨어 BOM + 권장 부품

  •    (C) 이벤트 룰(임계치) 표준안: 무활동/낙상/응급 키워드

  •    (D) 데이터 스키마(측정/이벤트/문진/알림) 초안

당신이 만들고 있는 형태(ESP32 기반인지, Pi급인지)에 맞춰 A안/ B안 중 하나로 확정해 더 구체
화해 드릴게요.

UserHelp --> OpenCase : 케이스 생성\n(urgent)

    RetryNetwork --> OpenCase : 재연결 성공\n(이벤트 업로드)

    RetryNetwork --> Escalate119 : 재연결 실패\n+ 위험도 CRITICAL

}

    [*] --> Stage1_Guardian1

    Stage1_Guardian1 --> Guardian1_Ack : 보호자 1 확인(ACK)

    Stage1_Guardian1 --> Stage2_Guardian2 : 타임아웃/실패

    Stage1_Guardian1 --> Escalate119 : 즉시 119 조건\n(CRITICAL)

 0) 공통 설계 규칙

  •    PII 최소화: user_id는 내부 UUID, 전화번호/주소 등은 별도 PII Vault 테이블로 분리

  •    Event-driven: “측정(시계열)”과 “이벤트(상태/경보)”를 분리

  •    감사로그 필수: 민감 데이터 조회/수정/조치 전부 남김

  •    보관정책(예시)

         o   raw 음성/원문 대화: 저장 안 함(또는 7일 이하, 동의 기반)

         o   측정 시계열: 1~3년(기관 정책)

         o   이벤트/알림/조치 기록: 3~5년(분쟁 대비)




 1) 엔터티 개요

  •    User: 대상자(독거노인)

  •    Device: 설치된 디바이스

  •    Measurement: 생체/활동 등 시계열 데이터

  •    Event: 낙상/무활동/응급/센서오류 등 “사건”

  •    Triage(문진): 원격진료/상담 전 문진

  •    Alert/Notification: 알림 발송 및 수신자별 딜리버리 상태

  •    Case/Incident: 하나의 사건을 “처리” 관점으로 묶은 티켓

  •    Audit Log: 접근/변경/조치 이력

 2) 이벤트→DB 레코드 생성 규칙 (매핑 규칙)

 아래는 룰 엔진/워크플로우 엔진이 이벤트를 받았을 때, 어떤 테이블에 어떤 레코드를 만들고
상태를 어떻게 바꾸는지에 대한 “구현 규칙”입니다.

 2.1 용어

  •   Incoming Event: 엣지/클라우드가 생성한 care_event

  •   Case Merge Window: 케이스 병합 시간창(기본 30분)

  •   Event Group: FALL, INACTIVITY, VITALS, DEVICE 등 (분류용)

  •   Stage: 콜 트리 단계(1~4, 119는 5)




 2.2 공통 규칙: 이벤트 수신 시

 Rule-0: care_event 생성(이미 생성되어 들어온 경우는 생략)

  •   입력: edge_rule/cloud_rule/manual

  •   작업:

          1.   care_event INSERT (status='open')

          2.   audit_log에 “EVENT_CREATE”
Rule-1: 케이스 병합/생성

조건

 •   동일 user_id에 대해,

 •   동일 event_group이고,

 •   최근 case_merge_window(30분) 내에 care_case.status in ('open','in_progress')가 있으면 병
     합

병합

 •   care_case.linked_event_ids에 event_id append

 •   case_action INSERT (action_type='LINK_EVENT', meta={event_id})

신규 생성

 •   care_case INSERT (status='open', priority=normal/high/urgent)

 •   care_case.linked_event_ids = [event_id]

 •   case_action INSERT (action_type='OPEN_CASE', meta={event_id, reason})

우선순위 매핑 예:

 •   INFO/WARN 이벤트 → priority=normal/high

 •   ALERT/CRITICAL 이벤트 → priority=high/urgent




2.3 “알림 생성” 규칙: alert / notification_delivery

Rule-2: alert 생성

케이스가 열리거나(또는 병합되었더라도) “알림이 필요”한 조건이면:

 •   alert INSERT

         o   case_id, event_id, user_id

         o   alert_type = 이벤트군/세부유형 매핑

         o   severity = 이벤트 severity

         o   status='pending'

         o   context에 아래 필드를 표준화해서 넣기:

                 ▪   {"stage":1,"policy_version":"2026.03","cooldown_key":"user+group","evidenc
                     e":{...}}
알림 필요 조건(기본)

 •   severity in ('ALERT','CRITICAL') → 항상

 •   severity='WARN' → “사용자 확인 실패/패턴 이상”일 때만

Rule-3: notification_delivery 생성(수신자/채널)

alert가 생성되면 정책에 따라 수신자들을 계산해 N건 생성:

 •   notification_delivery INSERT N건

         o   alert_id

         o   recipient_type = guardian/caregiver/operator

         o   recipient_id (내부 계정 있으면)

         o   channel = push/sms/voice_call

         o   send_status='queued'

채널 폴백 규칙

 •   push 실패 or 미확인 타임아웃 → sms/전화로 자동 전환

 •   전환 시 새 delivery 생성(또는 기존 delivery 업데이트) +
     case_action(action_type='ESCALATE_CHANNEL')




2.4 ACK/조치/종결: case_action 중심

Rule-4: 수신자 ACK 처리

 •   alert.status = 'acknowledged'

 •   care_event.status = 'acknowledged' (해당 사건)

 •   care_case.status = 'in_progress'

2.6 해결/종결 규칙

Rule-8: Resolved

안전 확인 또는 조치 완료 시:

 •   care_event.status='resolved' (관련 이벤트들 일괄 처리 가능)

 •   care_case.status='closed', closed_at=now()

 •   alert.status='acknowledged' 또는 expired 정리

 •   case_action INSERT (action_type='RESOLVED', meta={result:"safe_confirmed"})

Rule-9: False Positive

오탐이면:

 •   care_event.status='false_positive'

 •   care_case 종결(필요시)

  5) 구현 팁: “즉시 반영”에서 꼭 고려할 것

      •       정책 캐시 무효화: policy_bundle.version이 바뀌면 룰 엔진 캐시 flush
  •   정책 적용 스코프 우선순위: device/user/tenant 순으로 override

  •   스냅샷 기록: 사건 처리 시작 시점의 policy_version을 alert.context.policy_version에 저장
      (사후 감사)

  •   안전장치(Guardrail): 운영자가 임계치를 잘못 낮추면 알림 폭주 가능
      → 저장 시 “충돌/폭주 검사(예: cooldown=0 & severity=ALERT)” 같은 검증 룰을 추가




원하시면 다음 단계로:

  •   위 JSON 을 저장/검증/배포하는 Policy API 설계(REST endpoint + 권한 RBAC)

  •   policy_threshold → “실시간 평가 엔진”의 평가 순서/우선순위/단락(short-circuit) 로직

  •   실제 운영 콘솔에서 쓰는 정책 편집 UI 필드 모델(폼 스키마)
아래는 요청한 3 가지를 한 번에 구현 가능한 수준으로 정리한 설계안입니다.

     •   (1) Policy API 설계 (REST + RBAC + 검증/배포 플로우)

     •   (2) policy_threshold 기반 실시간 평가 엔진 평가 순서/우선순위/단락 로직

     •   (3) 운영 콘솔용 정책 편집 UI 필드 모델(폼 스키마)




1) Policy API 설계 (REST endpoint + RBAC)

1.1 역할/권한(RBAC) 모델

Roles

2) policy_threshold 기반 실시간 평가 엔진 로직

2.1 입력 스트림

   •    measurement 이벤트(시계열): (user_id, m_type, value, ts, quality, source)

   •    care_event 후보: (event_type, severity, confidence, ts, details)

   •    device_state 이벤트: online/offline/battery 등

   •    interaction 이벤트: 사용자 응답/미응답(“괜찮음/도움/무응답”)

2.2 평가 순서(권장 파이프라인)

원칙: 더 위험한 것부터, 계산이 싼 것부터, 폭주를 막는 것부터.

   1.   Guardrail 전처리

            o   정책 버전 로드 / 바인딩 적용(tenant→user→device 우선)

            o   쿨다운 체크(동일 그룹/동일 키)

            o   측정 quality/min_quality 필터

            o   시간대 예외(수면시간 등) 적용

   2.   Hard Rule(즉시 119) 평가            (가장 먼저)

            o   policy_rule.domain='EMS' + priority 오름차순

            o   참이면 short-circuit: EMS 액션 실행 (단, “병행 알림”은 허용)

   3.   CRITICAL threshold 평가

            o   policy_threshold.severity='CRITICAL'를 먼저 평가

            o   참이면 event_type 생성 + escalation_plan 시작
   4.   ALERT threshold 평가

           o    alert 생성 조건을 만족하면 event 생성

   5.   WARN/INFO 평가

           o    “사용자 확인 실패” 같은 2차 조건이 있을 때만 케이스/알림 유발

           o    아니면 단순 기록

   6.   Complex Rule(VERIFY/ESCALATION/VOICE)

           o    예: “낙상 후보 이벤트가 있고 60초 미응답이면 CRITICAL로 승격”

           o    이미 열린 케이스가 있으면 업데이트/승격




2.3 우선순위/단락(short-circuit) 규칙

(A) 단락 기준

   •    EMS immediate 룰이 true면:

           o    contact_ems(immediate)는 실행하고,

           o    동시에 보호자/관제 알림은 “병행”(parallel)로 둘 수 있음

   •    동일 event_group에서 이미 care_case.priority='urgent'로 진행 중이면:

           o    더 낮은 severity(WARN/INFO)는 평가 중단 또는 “기록만”

   •    같은 threshold key가 최근 lookback 내에 이미 발동했고 cooldown 중이면:

           o    알림 생성 중단, event는 선택적으로 “suppressed”로 기록(권장)

(B) 평가 우선순위 정의

   •    policy_rule.priority : 낮을수록 먼저

   •    policy_threshold는 다음 정렬 권장:

           1.   severity (CRITICAL>ALERT>WARN>INFO)

           2.   domain 우선순위(EMS>VITALS>FALL>INACTIVITY>DEVICE)

           3.   key 알파(안정적 순서를 위해)

   •    한 metric에 여러 임계치가 있으면:

           o    CRITICAL 성립 시 ALERT/WARN은 스킵
2.4 “지속/연속” 조건 처리(lookback/duration/occurrences)

      •    duration: 최근 duration_sec 동안 조건이 유지되는지 확인

      •    occurrences: lookback_sec 내 조건을 만족한 샘플 수(또는 연속 n회)

      •    구현:

                  o   사용자×metric별 ring buffer(메모리 캐시) 유지

                  o   또는 Timescale continuous aggregate 활용(대규모 시)

2) 룰 엔진 평가 엔진 의사코드 + 캐시 구조(tenant/user/device override)

2.1 캐시 구조(오버라이드 우선순위)

정책 결정은 device > user > tenant 순으로 override(또는 binding.priority 기반).
런타임에서 “어떤 bundle이 적용되는지”를 빠르게 찾는 캐시가 중요합니다.
캐시 레이어

    •    policyIndexCache[tenant_id]

             o   bindings 목록(정렬된 priority)

             o   active bundle 기본값

    •    effectiveBundleCache[(tenant_id, device_id, user_id, scope)] -> EffectivePolicy

             o   thresholds (정렬/인덱싱)

             o   rules (domain+priority 인덱싱)

             o   escalation plans (event_group+severity_min 인덱싱)

             o   etag/version/hash

인덱스(빠른 평가용)

    •    thresholdByMetric[metric] -> list[Threshold] (severity desc, domain priority)

    •    thresholdCriticalFirst = list[Threshold] (pre-sorted)

    •    rulesByDomain[domain] -> list[Rule] (priority asc)

    •    planResolver(event_group, severity) -> EscalationPlan + stages




2.2 EffectivePolicy 선택 로직(오버라이드 해석)

resolve_bundle(tenant_id, device_id, user_id):

  candidates = []

  for binding in policy_binding where tenant_id=tenant_id:

        if binding.scope_type == 'device' and binding.scope_id == device_id:
candidates.add(binding)

        if binding.scope_type == 'user'     and binding.scope_id == user_id:
candidates.add(binding)

        if binding.scope_type == 'tenant' (or scope_id null):
candidates.add(binding)

  sort candidates by priority asc, created_at desc

  if candidates not empty: return candidates[0].bundle_id

  else: return tenant_default_active_bundle
2.3 평가 엔진 pseudocode (short-circuit 포함)

on_signal(signal):

  # signal: measurement | candidate_event | device_state | interaction




  ctx = make_context(signal)

  # ctx includes: tenant_id, user_id, device_id, ts, scope(cloud/edge), recent_state_cache




  policy = get_effective_policy(ctx)

  # from effectiveBundleCache, reload if version changed




  # 0) Guardrails

  if signal.type == "measurement":

       if signal.quality < policy.min_quality_for(signal.metric): return RECORD_ONLY

       if is_suppressed_by_cooldown(ctx, policy, signal): return RECORD_ONLY

       update_ringbuffer(ctx.user_id, signal.metric, signal)   # for duration/occurrences




  if is_quiet_hours(ctx, policy) and policy.has_night_exception(signal):

       apply_night_adjustments(ctx, policy, signal)




  # 1) Evaluate HARD rules first (EMS immediate)

  for rule in policy.rulesByDomain["EMS"]:

       if rule.match_scope not in [ctx.scope, "both"]: continue

       if eval_rule(rule.rule_json, ctx):

            execute_actions(rule.action_json, ctx, policy)

            # short-circuit: EMS immediate typically ends evaluation of lower severity,

            # but may allow parallel notifications
         if action_json includes contact_ems(mode="immediate"):

                return DONE_WITH_EMS_PARALLEL_OK




# 2) Evaluate CRITICAL thresholds

for th in policy.thresholdsCriticalFirst:

    if th.enabled is false: continue

    if th.match_scope and ctx.scope mismatch: continue (optional)

    if th.metric not relevant to signal: continue

    if eval_threshold(th, ctx):     # includes duration/occurrences/lookback

         event = emit_or_update_event(th, ctx)      # care_event

         case    = open_or_merge_case(event, ctx, policy)

         plan    = resolve_plan(policy, event_group(event), event.severity)

         start_or_continue_escalation(case, event, plan, ctx, policy)

         # short-circuit: once CRITICAL fired for this group, stop lower checks

         mark_group_block(ctx.user_id, event_group(event), level="CRITICAL")

         return DONE




# 3) Evaluate ALERT thresholds

for th in policy.thresholdsAlert:

    if group_blocked(ctx.user_id, event_group_from_metric(th.metric), "CRITICAL"): continue

    if eval_threshold(th, ctx):

         event = emit_or_update_event(th, ctx)

         case    = open_or_merge_case(event, ctx, policy)

         plan    = resolve_plan(policy, event_group(event), event.severity)

         start_or_continue_escalation(case, event, plan, ctx, policy)

         # optionally continue to check other ALERT of different groups

         continue
 # 4) Evaluate WARN/INFO thresholds (only if needs escalation)

 for th in policy.thresholdsWarnInfo:

      if eval_threshold(th, ctx):

           event = emit_or_update_event(th, ctx)

           if needs_user_verify(event, policy, ctx):

                   schedule_device_verify(event, ctx)   # "괜찮으세요?" prompt

           else:

                   record_event_only(event)

           continue




 # 5) VERIFY / ESCALATION / VOICE rules (upgrade logic)

 for rule in policy.rulesByDomain["VERIFY"]:

      if eval_rule(rule.rule_json, ctx):

           execute_actions(rule.action_json, ctx, policy)




 for rule in policy.rulesByDomain["ESCALATION"]:

      if eval_rule(rule.rule_json, ctx):

           execute_actions(rule.action_json, ctx, policy)




 return DONE




핵심 함수 동작(요약)

  •    eval_threshold(th, ctx)

            o      metric ringbuffer에서 lookback/duration/occurrences 충족 확인

            o      operator 비교(LT/GTE/BETWEEN 등)
   •    emit_or_update_event(th, ctx)

            o   동일 이벤트가 열려 있으면 update(상태/증거)

            o   아니면 care_event INSERT

   •    open_or_merge_case(event, ctx, policy)

            o   merge_window(기본 30분) 내 동일 group 케이스 존재 시 병합

   •    start_or_continue_escalation(case, event, plan, ctx, policy)

            o   Stage1 alert 생성 → delivery 생성

            o   타임아웃 스케줄러 등록(예: Redis delayed job)




2.4 타임아웃/에스컬레이션은 “스케줄러”가 담당

평가 엔진은 “발생 순간”만 다루고, **타임아웃 전이(Stage 상승)**는 별도 워커가 안정적입니다.

   •    워커 입력: alert_id(stage), deadline_ts

   •    deadline 지나면:

            o   ACK 여부 확인(alert.status != acknowledged)

            o   case_action(ESCALATE_STAGE)

            o   다음 stage alert + delivery 생성
